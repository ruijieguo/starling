# Statement Bus

## 功能定义

Statement Bus 是所有 Statement 写入与事件传播的唯一入口，任何组件不得绕过 Bus 直接操作存储（公理 I 硬约束）。Bus 执行四项职责：(1) 通过 Validator 校验写入合法性，(2) 通过 ConflictProbe 探测冲突并建立边，(3) 通过 outbox 分发 BusEvent 给下游 Subscriber，(4) 通过幂等键与 causation_chain 防止事件风暴与重复副作用。Bus 不处理业务逻辑、检索路由或调度编排；这些责任分别属于 [EngramStore](v23_06_engramstore.md)、[Retrieval](v23_13_retrieval.md) 和 [Replay](v23_10_replay.md)。

三条写入入口：

- `append_evidence(engram_input)` — 创建 Engram，执行 source trust、权限、retention 与幂等校验，提交后 emit `evidence.appended`。
- `write(stmt)` — 写 Statement / 边 / 索引，同事务追加 outbox 事件。
- `rebuild_container(container_ref, sources)` — 重建 Persona / CommonGround 等物化视图，用 CAS 防并发覆盖。

---

## 输入

- Statement 写入请求（含 holder、subject、predicate、object、provenance）
- Engram 写入请求（含 retention_mode、ingest_policy）
- Container rebuild 请求（含 dimension key + version CAS）
- 外部 Subscriber 订阅注册

## 输出

- 持久化 commit 确认（含 stmt_id / engram_ref / container_version）
- BusEvent 流（按 outbox_sequence 顺序投递到 subscriber）
- Validator 拒绝原因（含错误码 + 详情）
- ConflictProbe 冲突报告（含 conflict_kind + canonical_conflict_key）

---

## 主要流程

### 1. 写入路径（append_evidence / write）

```
append_evidence(input)
  → EvidenceValidator (source_boundary check)
  → EngramStore (事务写 Engram)
  → outbox.append(evidence.appended)          ← 同事务提交
  → [异步] Extractor 消费 evidence.appended
      → Validator (schema / 隐私 / extraction一致性)
      → ConflictProbe (canonical_conflict_key 查询)
          ├─ direct_contradiction  → CONFLICTS_WITH 边 + emit belief.conflict
          ├─ superseding           → valid_to 截断 + SUPERSEDES 边 + emit statement.superseded
          ├─ partial_overlap       → CONFLICTS_WITH 边 + REVIEW_REQUESTED
          └─ adjacent              → TEMPORAL/ADJACENT 边
      → 主表 commit (Statement + 边 + canonical_object_hash 派生列)
      → outbox.append(statement.written | statement.derived | extraction.failed ...)
          ← 同事务提交
  → [异步] Dispatcher → Subscribers
```

抽取失败补偿路径：

```
Validator 拒收 →
  emit extraction.failed(failure_kind, engram_ref, raw_output_hash, retry_count)
    ├─ parse_error / schema_invalid  → 最多自动重试 2 次 + emit extraction.retry_scheduled
    ├─ low_confidence                → 候选 Statement(review_status=REVIEW_REQUESTED) 或 review 队列
    ├─ privacy_violation             → 不重试 + dead-letter + audit
    └─ 超过重试阈值                  → emit extraction.dead_lettered
                                       Engram.extraction_status = FAILED
```

自污染防护：`system_internal / observer_agent / replay_output` 类 source 默认 `NO_STORE`，只写 audit counter，不得创建普通 Engram，不得参与 holder 画像、Replay 或 ToM。

### 2. Container 重建（rebuild_container CAS 流程）

```
Bus.rebuild_container(container_id, dimension, expected_version, source_refs)
  → 校验 dimension 属于 §3.6 声明合法 key
      └─ 不合法 → emit container.dimension_invalid + 拒绝
  → P1: 整体 version CAS
      expected_version == current_version ?
        ├─ 是 → 沿 SUPERSEDES 链替换 stale source_refs → 写 Container → version+1
        │         → emit container.rebuilt
        └─ 否 → 指数退避重试；3 次后进入 Sleep 批处理
  → P3: dimension_versions CAS
      expected_version == dimension_versions[dimension] ?
        ├─ 是 → 只重建冲突 dimension → dimension_versions[dimension]+1
        │         → 更新 dimension_sequences[dimension]
        │         → emit container.rebuilt
        └─ 否 → 只重试冲突 dimension
  → Retrieval 若发现 active source_refs 含已 supersede Statement
      → 降级直接查 Statement 主表
      → 异步触发 Bus.rebuild_container 修复
  → 若最新版本 review_status=PENDING_REVIEW/REJECTED
      → active 视图保留旧 approved ref
      → payload 标 stale_due_to_pending_review=true
```

### 3. Reconsolidation 触发收敛（5 条路径）

所有路径最终写入同一 `pending_evidence` 队列；同 stmt_id 在可塑窗口期只允许一个活跃窗口，新触发追加证据不开新窗口。

```
触发路径                         事件                           防抖键
─────────────────────────────────────────────────────────────────────────────
1. Retrieval 命中               statement.recalled             (stmt_id, query_window_2s)
2. derived_from 引用已存在 stmt  statement.references_existing  (target_stmt_id, source_stmt_id, window_10s)
3. ConflictProbe 探针           belief.conflict                (canonical_conflict_key, window_10s)
4. 显式 API 触发                reconsolidate.requested        (stmt_id, request_id)
5. Commitment 终态              commitment.fulfilled/broken    (commitment_id, terminal_state)
─────────────────────────────────────────────────────────────────────────────
→ 全部追加到 pending_evidence 队列
→ 窗口 close → Reconsolidation Engine 统一仲裁  →  参见 v23_11_reconsolidation.md
```

### 4. severe path 双路径（local-store vs dist-store）

```
Bus.write(stmt, severity="severe")
  ├─ local-store 部署
  │     → 原子事务（Statement + CONFLICTS_WITH 边 + outbox 同一 SQLite 事务）
  │     → commit 后异步投递
  └─ dist-store 部署
        → saga 补偿模式
            step1: 写 Statement 主表（Neo4j / 分布式存储）
            step2: 写 CONFLICTS_WITH 边（可能跨分片）
            step3: append outbox 事件
            任一 step 失败 → 补偿事务回滚已成功 step
                           → emit system.delivery_failed
                           → 进入 dead-letter queue
```

## 核心算法

### 1. canonicalize_object

将 Statement.object 标准化为确定性可哈希表示，禁止 list / dict 兜底 hash。

| 输入类型 | 标准化规则 |
|---|---|
| `bool` | 统一为 Python `True` / `False`（不做 int 转换） |
| `int` | 原值，禁止 float 强转 |
| `float` | 规范化精度（去尾部零），禁止 NaN / Inf |
| `str` | Unicode NFC 归一化，去首尾空白 |
| `datetime` | UTC ISO-8601，精度统一到秒 |
| `EntityRef` / `CognizerRef` | `canonical_name + ":" + id`（不展开内部属性） |
| `StatementRef` | 一阶展开（见下） |
| `list` / `dict` | 不允许兜底 hash；调用方必须先拆解为标量或 Ref |

**StatementRef 一阶展开**：

```python
canonicalize_object(StatementRef) = hash(
    inner.holder,
    inner.subject,
    inner.predicate,
    inner.polarity,
    inner.modality,                      # 区分 BELIEVES vs DOUBTS
    canonicalize_object(inner.object),   # 递归一层；inner.object 若仍为 StatementRef 则只取 id_hash，不再递归
)
```

只展开一层：二阶 ToM 冲突检测准确；三阶以上受 [ToM](v23_09_tom.md) `derived_depth ≤ 3` 限流保护。

### 2. canonical_object_hash

Statement 写入时由 `canonicalize_object(stmt.object)` 计算并持久化为主表派生列，禁止查询时运行时重算。

相关字段：

- `canonical_object_hash`：存储当前计算结果。
- `canonical_object_hash_version`：默认 `"v1"`，记录 canonicalize 规则版本。
- ConflictProbe 只对同版本 hash 做直接比较；跨版本命中最多降级为 `partial_overlap + REVIEW_REQUESTED`。

**hash version 升级期协议**：

```
canonicalize 规则升级（新版本号如 "v2"）
  → 旧行保持原版本，后台 maintenance 异步 rebuild
  → 进入 hash-mixed 区间期间二选一：
      双查协议：新写入同时计算当前版本 hash 与上一版本 compatibility hash
               → 只有同版本命中才允许 direct_contradiction / superseding
               → 跨版本命中只能标 REVIEW_REQUESTED
      保守协议：无法计算上一版本 hash 时
               → 该 key 范围内所有新写入只能落 REVIEW_REQUESTED
               → 不得自动判定无冲突
  → maintenance 超过 14 天未完成
      → emit projection.upgrade_overdue（Consumer=Ops）
      → RuntimeHealth 进入 DEGRADED
  → rebuild 完成 → emit projection.upgrade_completed → RuntimeHealth 回 READY
```

设计理由：双查热路径成本 2×；14 天上限确保 Ops 可见性，避免长期 2× 负载被忽略。

### 3. ConflictProbe

写入前查询候选集，按冲突等级建边并 emit 事件，不静默覆盖。

| 等级 | 判定 | 处理 |
|---|---|---|
| `direct_contradiction` | 同 holder / modality / subject / predicate / scope，时间区间等价或高度重叠，polarity 相反 | emit `belief.conflict`，进入 Reconsolidation |
| `partial_overlap` | canonical object 相同，时间区间有交集但不完全覆盖 | 建 `CONFLICTS_WITH` 边，降低自动仲裁置信度，Context Pack 标 CONFLICT |
| `superseding` | 新 Statement valid interval 完全覆盖旧 Statement，且 evidence 更新 | 旧版 valid_to / expired_at 截断，建 `SUPERSEDES` 边，通常不进入冲突流程 |
| `adjacent` | 时间区间相邻或存在小 gap，语义兼容 | 建 TEMPORAL / ADJACENT 边，作为补充而非冲突 |

时间 overlap 使用闭开区间 `[valid_from, valid_to)`，缺失 valid_to 视作 open-ended。overlap 查询条件：`start <= query_end AND (end > query_start OR end IS NULL)`。

若 event_time / valid_from 来自低置信 LLM 推断，ConflictProbe 不做 `direct_contradiction`，最多标 `partial_overlap + REVIEW_REQUESTED`。`UNKNOWN_INTERVAL` 只能参与低置信 partial overlap，不得触发 direct_contradiction。

**canonical_conflict_key 七元组**：

```python
canonical_conflict_key(stmt) = hash_tuple(
    stmt.holder,
    stmt.modality,
    stmt.subject,
    stmt.predicate,
    canonicalize_object(stmt.object),
    normalize_interval(stmt.valid_from, stmt.valid_to, stmt.event_time),
    scope_of(stmt),   # Norm: scope.kind+sorted(members); Commitment: (principal,beneficiary);
                      # CommonGround: sorted(parties); 普通 Statement: NULL
)
```

`object`、时间区间与 scope 必须参与 key，否则不同 object / deadline / 群组的 Statement 会被误合并或误判冲突。

**CommonGround supersede 触发序列**：

1. ConflictProbe 在 Bus.write 事务内识别 `superseding`，写 `SUPERSEDES` 边并 emit `statement.superseded`。
2. CommonGround Builder 消费 `statement.superseded`；只有新 Statement 已达到 `grounded` 条件时才执行 `SupersedeGround`。
3. `Bus.rebuild_container(kind="common_ground")` 一致性校验可补漏。
4. Replay Scheduler 只能通过 Bus 产生 `statement.derived / statement.corrected`，不得直接修改 CommonGround（绕过 outbox、CAS 与审计）。

### 4. 防抖契约

```
同 canonical_conflict_key 在窗口 W（默认 10s）内触发的同类事件
  → 合并为一个事件（取最后一次 payload）
  → 避免冲突反复仲裁
```

**causation_chain 传播规则**：

- Subscriber 处理事件 `E` 时，把 `E.event_id` 作为 `causation_parent` 携带。
- 调用 `Bus.write / Bus.emit / Bus.append_evidence / Bus.rebuild_container` 时，必须显式传入 `causation_parent`；根因事件显式传 `None`，不允许省略。
- Bus 生成新事件 `N`：若有 parent，`N.causation_chain = parent.causation_chain + [parent.event_id]`；若 parent 为 None，链为空。
- 链长检查在 Bus 发布事件时执行：`len(causation_chain) > 3` 直接拒绝派生事件，emit `system.runaway(chain_root, depth, source_event_id)`，并在相关 `PipelineRun.warnings` 中记录。

**幂等键公式**：

```python
idempotency_key = hash(
    event_type,
    aggregate_id,
    canonical_key,
    causation_chain_root,
    window_bucket,
)
```

Bus outbox 与 Subscriber inbox 均持久化该 key；重复投递直接 ACK。inbox idempotency_key 保留期：7 天，超期清理，Subscriber 启动时从 outbox sequence checkpoint 恢复。

**业务幂等冻结默认值**：

P1 只冻结两个产生用户可见重复副作用的键：

| 事件 | 业务幂等键 | 窗口粒度 |
|---|---|---|
| `commitment.fire` | `(commitment_id, trigger_id, fire_bucket)` | 24h |
| `statement.recalled` | `(query_id, query_window)` | 2s |

其他事件默认使用一次性业务键 `(event_type, aggregate_id, version_or_sequence)`，其中 `version_or_sequence` 取 `payload.target_state` / `payload.dimension_version`（状态转换事件）或 `outbox_sequence`（非状态转换事件）。

**Dispatcher checkpoint**：`consumer_checkpoint(subscriber, shard, last_dispatched_sequence)` 持久化；重启后从 checkpoint+1 恢复扫描。Subscriber 处理成功后先写本地 inbox ACK / 业务幂等记录，再更新 checkpoint；ACK 丢失导致重复投递时按 inbox 去重并直接 ACK。

**冻结默认值汇总**：

| 参数 | 值 |
|---|---|
| causation_chain 深度上限 | 3 |
| 防抖窗口 W（belief.conflict / statement.references_existing） | 10s |
| 防抖窗口（statement.recalled） | 2s |
| 防抖窗口（commitment.fire） | 24h |
| inbox idempotency_key 保留期 | 7 天 |
| canonical_object_hash 升级 maintenance 上限 | 14 天 |
| MAX_CONSOLIDATION_ATTEMPTS | 5 |
| T_max_volatile | 7 天 |
| T_replay_consolidating_timeout | READY 1h / DEGRADED 4h |
| Commitment BROKEN 计数上限 | 3（达到自动 WITHDRAWN） |
| RENEGOTIATED supersedes 链长 | ≤3 |

### 5. extraction_span_key

用于 Extractor 重试去重的幂等键，Bus.write 对同一 key 做幂等 upsert：

```python
extraction_span_key = hash(
    engram_ref,
    chunk_index,        # P1；P3 替换为 (segment_id, span_start, span_end)
    normalized_span,
    predicate,
    canonical_object,
)
```

- P1：`normalized_span = chunk_index`；同 chunk 内多条 Statement 由 `(predicate, canonical_object)` 区分。
- P2 启用 `segment_map` 后：`normalized_span = (segment_id, span_start, span_end)`，字符级精确去重。
- 重试只覆盖失败片段，已成功片段不重新写入。
- 重试产出与已成功 Statement 同 span 但内容不同，进入 `REVIEW_REQUESTED`，不自动 supersede。

## 数据结构

### BusEvent envelope

```python
class BusEvent:
    event_id: UUID
    event_type: str                        # 见事件枚举表
    primary_id: str                        # 业务幂等默认键的主标识（producer 必须明确填写）
                                           # Statement 事件取 stmt_id；Container 事件取 container_id；
                                           # Evidence 事件取 engram_ref；
                                           # Conflict 事件取 canonical_conflict_key；
                                           # Pipeline 事件取 run_id
    aggregate_id: str                      # 同 aggregate 顺序投递；可等于 primary_id 或更粗粒度 id
    outbox_sequence: int                   # 单调递增；同 aggregate 内按此顺序投递
    causation_chain: list[str]             # list[event_id]；长度 ≤ 3
    idempotency_key: str                   # hash(event_type, aggregate_id, canonical_key,
                                           #       causation_chain_root, window_bucket)
    payload: dict
```

aggregate_id 默认表：

| 事件族 | primary_id | aggregate_id | 顺序语义 |
|---|---|---|---|
| `statement.*` | `stmt_id` | `stmt_id` | 单 Statement 生命周期顺序 |
| `belief.conflict` | `canonical_conflict_key` | `conflict_window_id` 或 `canonical_conflict_key` | 冲突窗口内顺序；consumer 必须补查相关 stmt 当前状态 |
| `statement.superseded / corrected` | `new_stmt_id` | `supersedes_root_id` | 同一版本链顺序 |
| `container.*` | `container_id` | `container_id:dimension` | 同一 Container dimension 顺序 |
| `evidence.*` | `engram_ref` | `engram_ref` | 同一 Engram retention / extraction 顺序 |
| `pipeline.*` | `run_id` | `run_id` | 同一长任务顺序 |
| `commitment.*` | `commitment_id` | `commitment_id` | 同一承诺状态机顺序 |

`belief.conflict` 事件可能早于某些相关 `statement.*` 事件被 subscriber 看见；Reconsolidation / Container Builder 消费冲突事件时必须按 stmt_id 补查主表，以主表事务状态为准，不能只相信事件到达顺序。

### ValidatorRule

```python
class ValidatorRule:
    # schema 必填字段
    required_fields: list[str] = [
        "tenant_id", "holder", "perspective", "modality",
        "polarity", "observed_at", "provenance", "review_status",
    ]
    # evidence / derived_from 分级
    provenance_rules: dict = {
        "user_input":               {"evidence": "≥1 EngramRef 必填", "source_spans": "≥1 必填", "derived_from": "可为空"},
        "tom_inferred":             {"evidence": "可为空", "source_spans": "不强制；若有只作 supporting", "derived_from": "≥1，除非只返回 transient context 不落库"},
        "replay_derived":           {"evidence": "可为空", "source_spans": "不强制；不得伪造直接 span", "derived_from": "≥1"},
        "reconsolidation_derived":  {"evidence": "可为空", "source_spans": "不强制", "derived_from": "≥1，必须指向被修正旧版 Statement"},
    }
    # tenant 派生规则
    tenant_derivation: str = "直接抽取 Statement 的 tenant_id 由 holder.tenant_id 派生，写入后不可变"
    # temporal_anchor 阶段要求
    temporal_anchor_p0: str = "P1 可为 None；序列化时若为空按 source_spans[0].observed_at 推导"
    temporal_anchor_p2: str = "P2 启用 segment_map 后，直接抽取 Statement 的 temporal_anchor 必填，Validator 拒绝 None"
```

**evidence / derived_from 分级规则（完整表）**：

| provenance | evidence 要求 | source_spans 要求 | derived_from 要求 |
|---|---|---|---|
| `user_input` | ≥1 条 EngramRef，必填 | ≥1，必填（P1 chunk 级即可） | 可为空 |
| `tom_inferred` | 可为空 | 不强制；若有只作 supporting | ≥1，除非只返回 transient context 不落库 |
| `replay_derived` | 可为空 | 不强制；不得伪造直接 span | ≥1 |
| `reconsolidation_derived` | 可为空 | 不强制 | ≥1，必须指向被修正旧版 Statement |

**跨 tenant 派生例外规则**：

| 前驱集合情况 | 处理 |
|---|---|
| 含 `tenant_id ∈ {"org_default","system"}` | 允许派生继承请求方 / holder tenant；audit metadata 标 `cross_tenant_derived=true` 与前驱 tenant 列表 |
| 多个 user tenant 且存在显式跨 tenant 协议 | 允许写入；`review_status=REVIEW_REQUESTED`，由策略或人工确认最终 tenant |
| 多个 user tenant 且无显式协议 | Validator 拒绝写入；emit `statement.review_requested` 或 `extraction.failed(schema_invalid)` |

`tenant_id` 写入后不可变；任何 UPDATE tenant_id 的路径都必须 fail-closed。

**holder_perspective 判定规则**（Extractor LLM prompt 必须显式包含）：

| 场景 | holder_perspective |
|---|---|
| 主语为 self（agent 自己）且 present_in_episode | `FIRST_PERSON` |
| 主语为他人，原文是直接引语或明确转述 | `QUOTED` |
| 转述但不能追溯到原始来源（听说 / 传闻） | `HEARSAY` |
| LLM 推断生成的填空证据（无原文支持） | `INFERRED` |

不允许默认值兜底；默认 INFERRED 会错误标记 FIRST_PERSON 例，污染 Context Pack。写入即冻结（同其他归因字段）。

**Extractor 产出上限与默认值**：

| 项 | 规则 |
|---|---|
| 产出上限 | 每条 Engram 默认最多 10 条 Statement；超过按 salience / stakes / explicitness 排序截断，剩余候选进入 review queue |
| confidence 初值 | first-person 明确陈述 ≥0.8；直接观察 0.7-0.9；hearsay 0.5-0.7；系统推断 ≤0.5；低于 0.3 不落库 |
| review 默认值 | holder / perceived_by / confidence 任一来自弱推断时，默认 `review_status=INFERRED_UNREVIEWED` |

### ConflictProbe 索引契约（P1 主表必须）

ConflictProbe 是写入路径的一部分，不属于 P2 Projection Index。SQLite 主表必须提供以下两个索引，保障每次写入前的候选查询不退化为全表扫描：

| 索引名 | 字段 | 用途 |
|---|---|---|
| `idx_conflict_lookup` | `(tenant_id, holder, modality, subject, predicate, canonical_object_hash_version, canonical_object_hash)` | 候选集前置过滤 |
| `idx_temporal_overlap` | `(tenant_id, holder, valid_from, valid_to)` | 时间区间 overlap 检测 |

`canonical_object_hash` 写入时计算并持久化为派生列，禁止每次查询时运行时重算。

### pending_evidence 队列结构

```python
class PendingEvidenceQueue:
    stmt_id: UUID                          # 关联的 Statement
    window_id: UUID                        # Reconsolidation 窗口 ID
    items: list[BusEvent]                  # 各触发路径追加的事件
    capacity: int = 100                    # 单窗口最大事件数（超出时按 salience 淘汰低优先级）
    eviction_policy: str = "lru_by_salience"  # 容量满时淘汰低 salience 事件
    opened_at: datetime
    window_close_at: datetime              # 窗口关闭时间；close 后触发统一仲裁
    status: Literal["open","closed","arbitrated"]
```

同 stmt_id 在可塑窗口期只允许一个 `status=open` 窗口；新触发事件追加到现有窗口的 items，不开新窗口。

### ExtractionAttempt（最小账本）

```python
class ExtractionAttempt(BaseEntity):
    id: UUID
    run_id: UUID                             # 所属 PipelineRun(kind="extraction")
    engram_ref: EngramRef
    attempt_no: int
    extractor_version: str
    prompt_version: str
    prompt_input_hash: str                   # 覆盖 EngramStore hash + chunk/span + extractor config + existing_ref_map + holder context
    existing_ref_map: dict[str, StatementRef]  # 短 id -> 真实 StatementRef；防 LLM 幻觉引用长 UUID
    status: Literal["RUNNING","SUCCESS","PARTIAL_SUCCESS","FAILED","DEAD_LETTERED","NOOP"]
    accepted_statement_ids: list[StatementRef]
    rejected_fragments: list[dict]           # span_key / failure_kind / reason / raw_hash
    noop_reason: Optional[str]               # 如 no_new_memory / all_duplicate / below_confidence
    raw_output_hash: Optional[str]
    started_at: datetime
    finished_at: Optional[datetime]
```

ExtractionAttempt 是 `PipelineRun(kind="extraction")` 的 item-level 明细，不是独立的任务生命周期。调度、重试、lease、dead-letter 与最终终态以 PipelineRun 为权威。

### Container（物化视图）

```python
class Container(BaseEntity):
    id: UUID
    kind: Literal["persona","common_ground","knowledge_frontier"]
    source_refs: list[StatementRef]            # 存储稳定 Statement.id，不是内容快照
    materialized_payload: dict
    version: int                               # P1 整体 CAS 锁
    dimension_versions: dict[str, int] = {}    # P3；P1 可为空，只用整体 version
    dimension_sequences: dict[str, int] = {}   # P3；每个维度已消费的最高 outbox sequence
    last_rebuilt_at: datetime
    build_policy: Literal["on_event","sleep","manual"]
```

`source_refs` 存储稳定 `Statement.id`，不是内容快照；`Bus.rebuild_container` 必须沿 `SUPERSEDES` 链检查每个 ref，将旧版本替换为最新未 FORGOTTEN 版本 id。

上述 Python 示例为绑定层接口契约。核心实现为 C++ 抽象类，Python/JS/Rust 等绑定通过 pybind11 / NAPI / cxx 自动生成存根。

## 相关概念

**canonicalize_object**
将 Statement.object 转换为确定性可哈希表示的函数。bool / int / float / str / datetime 按类型标准化；EntityRef / CognizerRef 取 canonical_name + id；StatementRef 做一阶展开；不允许 list / dict 兜底 hash。

**canonical_object_hash**
写入时由 `canonicalize_object(stmt.object)` 计算并持久化为主表派生列。禁止运行时重算。ConflictProbe 用此列做候选集前置过滤。

**canonical_object_hash_version**
记录 canonicalize 规则版本（如 `"v1"`）。ConflictProbe 只对同版本 hash 做直接比较；跨版本命中只能标 `partial_overlap + REVIEW_REQUESTED`。升级时用新版本号，旧行保持原版本，后台 maintenance 异步 rebuild。

**canonical_conflict_key**
冲突探测的标准化键，七元组：`(holder, modality, subject, predicate, canonicalize_object(object), normalize_interval(valid_from,valid_to,event_time), scope_of(stmt))`。同 key 在防抖窗口内的同类事件合并为一。

**causation_chain**
事件 envelope 中的字段，类型为 `list[event_id]`，记录事件的因果链。链长 ≤ 3；超过时 Bus 拒绝派生事件并 emit `system.runaway`。

**causation_parent**
Subscriber 处理事件时携带的上下文字段，用于标识当前事件的直接前驱。调用 Bus 写入时必须显式传入；根因事件传 `None`。

**causation_chain_root**
causation_chain 中第一个 event_id，即最早的根因事件。用于 idempotency_key 计算。

**idempotency_key**
`hash(event_type, aggregate_id, canonical_key, causation_chain_root, window_bucket)`。Bus outbox 与 Subscriber inbox 均持久化该 key，重复投递直接 ACK。inbox 保留期 7 天。

**window_bucket**
幂等键的时间分片字段，不同事件类型的粒度不同：`commitment.fire` 为 24h，`belief.conflict / statement.references_existing` 为 10s，`statement.recalled` 为 2s。

**aggregate_id**
用于同 aggregate 内顺序投递的字段；可等于 primary_id 或更粗粒度的 run / window id。Subscriber 不得假设跨 aggregate 顺序。

**outbox_sequence**
单调递增整数；同 aggregate 内按此顺序投递。Dispatcher checkpoint 记录 subscriber / shard 的 last acked sequence，重启后从 checkpoint+1 恢复。

**pending_evidence**
Reconsolidation 可塑窗口期内的事件缓冲队列。同 stmt_id 只允许一个 `status=open` 窗口；新触发事件追加到现有窗口；窗口 close 时统一仲裁。单窗口容量 100，超出时按 salience 淘汰低优先级事件。

**Container CAS**
`Bus.rebuild_container` 的乐观锁机制。P1 使用整体 `version` 字段；P3 使用 `dimension_versions[dimension]` 细粒度锁。冲突时指数退避重试；P1 重试 3 次后进入 Sleep 批处理，P3 只重试冲突 dimension。

**dimension_versions**
Container 字段，`dict[str, int]`，P3 启用时记录每个 dimension 的版本号，用于细粒度 CAS 锁。dimension 键必须属于 [§3 数据本体](../04_starling_design_v19.md) §3.6 声明的合法 key；非法 key 触发 emit `container.dimension_invalid`。

**testing helper marker**
在测试代码中标记辅助函数或数据构造器的标志，用于区分生产逻辑与测试基础设施。与 Bus 无直接交互，但 ExtractionAttempt 的 `existing_ref_map` 测试构造必须通过此标记隔离，防止测试短 id 污染生产 UUID 引用。

**evidence 分级（provenance 枚举）**

| 值 | 含义 |
|---|---|
| `user_input` | 来自用户直接输入；evidence 与 source_spans 必填 |
| `tom_inferred` | 由 [ToM](v23_09_tom.md) 推断生成；derived_from 通常必填 |
| `replay_derived` | 由 [Replay](v23_10_replay.md) 派生；derived_from 必填，不得伪造直接 span |
| `reconsolidation_derived` | 由 [Reconsolidation](v23_11_reconsolidation.md) 产生的修正版本；derived_from 必须指向被修正旧版 Statement |

**review_status 枚举**

| 值 | 含义 |
|---|---|
| `APPROVED` | 已审核通过，置信度达标 |
| `PENDING_REVIEW` | 等待人工或规则复核 |
| `INFERRED_UNREVIEWED` | 由弱推断产生，未复核 |
| `REVIEW_REQUESTED` | 需要人工确认（低置信 / 跨 tenant / 合规擦除影响） |
| `REJECTED` | 已拒绝；Container active 视图不展示此版本 |

**跨 tenant 派生例外规则**

| 前驱集合情况 | 处理 |
|---|---|
| 含 `tenant_id ∈ {"org_default","system"}` | 允许派生继承请求方 / holder tenant；audit metadata 标 `cross_tenant_derived=true` 与前驱 tenant 列表 |
| 多个 user tenant 且存在显式跨 tenant 协议 | 允许写入；`review_status=REVIEW_REQUESTED`，由策略或人工确认最终 tenant |
| 多个 user tenant 且无显式协议 | Validator 拒绝写入；emit `statement.review_requested` 或 `extraction.failed(schema_invalid)` |
