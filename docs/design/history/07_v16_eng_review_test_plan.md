## 1. 状态机契约用例（5 条 CRITICAL）

### TC-A1-001 [CRITICAL] VOLATILE↔REPLAYING_CONSOLIDATING 最大轮次
- **Given**: Statement 进入 VOLATILE，被 Replay 选中转 REPLAYING_CONSOLIDATING，但每次都判定"证据不足"返回
- **When**: replay_count 达到 MAX_CONSOLIDATION_ATTEMPTS（默认 5）
- **Then**: 该 Statement 被强制转 CONSOLIDATED 且 review_status=PENDING_REVIEW，emit `statement.consolidation_forced` 审计事件

### TC-A1-002 [CRITICAL] VOLATILE TTL 兜底
- **Given**: Statement 停留 VOLATILE 超 T_max_volatile（默认 7 天）且不在 Affect Buffer
- **When**: 定时清理任务运行
- **Then**: Statement 自动 decay 至 ARCHIVED，emit `statement.archived`，原因 `volatile_ttl_exceeded`

### TC-A5-001 [CRITICAL] REPLAYING_CONSOLIDATING fallback timeout
- **Given**: Statement 进入 REPLAYING_CONSOLIDATING 后 Replay Scheduler 崩溃
- **When**: 进入该状态 > 1 小时（READY 状态）或 > 4 小时（DEGRADED/DRAINING 状态）
- **Then**: 自动回退 VOLATILE，emit `statement.replay_timeout`

### TC-A6-001 [CRITICAL] T5/T8 并发互斥（事件总线串行化）
- **Given**: CONSOLIDATED Statement S，同一毫秒内 (1) Retrieval emit statement.recalled 触发 T5；(2) Replay decay 公式判定 S(t)<0.05 emit statement.decay_candidate
- **When**: outbox dispatcher 按 stmt_id aggregate 顺序处理
- **Then**: 后到的事件读到 state 已变，幂等跳过；只产生一次状态变更

### TC-A6-002 决定性顺序
- **Given**: 同上场景，但 statement.recalled 先入 outbox
- **When**: dispatcher 处理
- **Then**: T5 先执行（state 变 REPLAYING_RECONSOLIDATING），后到的 decay_candidate 检查 state 不是 CONSOLIDATED 即跳过

## 2. Commitment 契约用例（3 条 CRITICAL）

### TC-A2-001 [CRITICAL] BROKEN 计数自动 WITHDRAWN
- **Given**: 同 commitment_id 进入 BROKEN 状态 3 次
- **When**: 第 4 次 deadline 过
- **Then**: 不转 BROKEN，自动 WITHDRAWN，emit `commitment.auto_withdrawn(reason=chronic_failure)`，trust_priors 下调

### TC-A2-002 [CRITICAL] RENEGOTIATED 链长上限
- **Given**: Commitment supersedes 链长 = 3
- **When**: 第 4 个 RENEGOTIATED 尝试
- **Then**: 拒绝，要求先 WITHDRAWN 再重新 COMMIT，emit `commitment.renegotiation_blocked`

### TC-A9-001 [CRITICAL] 未结清 Commitment 保护 Statement
- **Given**: Commitment(stmt_id=X) 状态 ACTIVE，emit `commitment.active_holding(X)`，decay scheduler 持有 in-memory set 含 X
- **When**: Replay decay 检查 X 是否可 ARCHIVED
- **Then**: X 跳过 decay（in-memory set O(1) 检查命中）

### TC-A9-002 终态释放
- **Given**: Commitment(X) 进入 FULFILLED，emit `commitment.released(X)`
- **When**: decay scheduler 处理 commitment.released
- **Then**: X 从 in-memory set 移除，下次 decay 候选可包含 X

## 3. Reconsolidation 契约用例（3 条 CRITICAL + 1 条）

### TC-A3-001 derived_from 并发防抖
- **Given**: 5 条新 Statement 同时通过 derived_from 引用同一旧 Statement Y
- **When**: 全部 emit statement.references_existing(target=Y)
- **Then**: Reconsolidation 对 Y 只开一个窗口，5 条引用进入同一 pending_evidence 队列

### TC-Q1-001 statement.references_existing 闭环
- **Given**: 新 Statement Z 写入，derived_from=[Y]，Y 状态 CONSOLIDATED
- **When**: Bus.write 完成
- **Then**: emit `statement.references_existing(source=Z, target=Y, kind=supporting|contradicting)`，Reconsolidation Engine 接收

### TC-A8-001 [CRITICAL] severe path 同事务原子提交（local-lite）
- **Given**: Reconsolidation 仲裁 contradict-severe，需提交 4 项：新 Statement、supersedes 边、旧 stmt ARCHIVED、3 条 outbox 事件
- **When**: SQLite 事务执行，模拟其中第 3 步抛异常
- **Then**: 整个事务回滚，旧 stmt 保持 CONSOLIDATED，新 Statement 不存在，outbox 无事件，audit log 记录失败

### TC-A8-002 [CRITICAL] severe path saga 补偿（cloud-graphiti）
- **Given**: cloud-graphiti profile 跨 Postgres+Neo4j+Qdrant 三库
- **When**: 第 2 步（Neo4j supersedes 边写入）失败
- **Then**: 触发 saga 补偿：撤销第 1 步（删除新 Statement），emit `reconsolidation.compensated`，旧 stmt 保持 CONSOLIDATED

### TC-Q3a-001 [CRITICAL] mild correction provenance 不变
- **Given**: user_input Statement S，confidence=0.9
- **When**: mild correction（轻微反对）触发 confidence 下调至 0.7
- **Then**: S.provenance 仍是 user_input（未变），confidence_history 增加一条记录 [(0.9, ts1), (0.7, ts2)]

## 4. Extractor 契约用例（4 条 CRITICAL）

### TC-Q2-001 [CRITICAL] holder_perspective = FIRST_PERSON
- **Given**: Episode 主语为 self（agent 自己），present_in_episode=True
- **When**: Extractor 抽取 Statement
- **Then**: holder_perspective = FIRST_PERSON

### TC-Q2-002 [CRITICAL] holder_perspective = QUOTED
- **Given**: 原文形式："Alice 说 'Bob 不再负责 auth'"
- **When**: Extractor 抽取 holder=Bob 的相关 Statement
- **Then**: holder_perspective = QUOTED，source_speaker=Alice

### TC-Q2-003 [CRITICAL] holder_perspective = HEARSAY
- **Given**: 原文形式："我听说 Carol 接手了 auth"，无原始来源
- **When**: Extractor 抽取
- **Then**: holder_perspective = HEARSAY

### TC-Q2-004 holder_perspective = INFERRED
- **Given**: LLM 填空生成的推断证据
- **When**: Extractor 抽取
- **Then**: holder_perspective = INFERRED

## 5. ConflictProbe 契约用例（1 条）

### TC-Q3b-001 二阶 Statement object 区分
- **Given**: 两条 Statement
  - S1: holder=A, modality=BELIEVES, subject=B, object=StatementRef("likes_coffee")
  - S2: holder=A, modality=BELIEVES, subject=B, object=StatementRef("likes_tea")
- **When**: ConflictProbe 计算 canonical_object_hash
- **Then**: S1 和 S2 哈希不同，不被误判为冲突合并

## 6. 默认值与事件表契约（3 条）

### TC-A4-001 system.runaway 事件 Consumer
- **Given**: causation_chain 深度达到 4
- **When**: Bus 检测到
- **Then**: emit `system.runaway`，Ops 和 Audit 同时接收

### TC-A7-001 默认值 baseline
- **Given**: 全新部署，未配置任何参数
- **When**: 启动
- **Then**: Online N=3, Idle T=120s, inbox 保留 7 天, window_bucket=24h

### TC-A7-002 window_bucket 粒度
- **Given**: 同一 commitment_id 不同时间窗发生 fire 事件
- **When**: 24h 内重复 fire
- **Then**: 幂等键命中，仅注入 Working Set 一次
