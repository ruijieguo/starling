# Starling Memory 实现路线图（Roadmap Index）

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement each milestone plan task-by-task.
> 本文件是**索引**，不直接执行。每个里程碑配独立 plan 文件，按下表入口找。

**Goal:** 按 P1（最小闭环）→ P2（社会心智 + 类脑动力学 + 前瞻）→ P3（完整 Retrieval Planner + 二阶 ToM + 多底座产品化）三阶 32 周实现 Starling Memory。

**Architecture:** C++20 内核 + pybind11 / NAPI / cffi 多语言绑定；三档 Substrate Profile（local-store seekdb+LadybugDB → dist-store Postgres+pgvector+AGE → cloud-store 三形态）；Statement Bus + outbox 单一写入入口；六态 ConsolidationState 状态机驱动 Hippocampus / Neocortex 双库分区。

**Tech Stack:**
- 内核：C++20（CMake + Ninja）
- 绑定：pybind11（Python 优先）、NAPI（JS）、cffi（Rust/C）
- 存储 P1：seekdb 1.2（关系 + 向量 + FTS）+ LadybugDB（嵌入式图）
- 存储 P2：Postgres 18 + pgvector + Apache AGE
- 存储 P3：单云原生 / 混合托管 / 跨云 SaaS
- 测试：GoogleTest（C++）+ pytest（Python）+ clang-tidy + ASan/UBSan
- LLM：通过 Adapter 抽象（OpenAI / Anthropic / 本地兼容）
- 评测：LongMemEval / ToMBench / FANToM / SoMi-ToM

---

## 为什么本文件不是单一巨型 plan

writing-plans skill 的硬约束："Each task should produce self-contained changes". 本工程 12 子系统 × 32 周，把所有里程碑塞一份文件等价于在不知道契约具体形态时凭空写步骤；后续里程碑的 schema、API、测试都依赖前一里程碑落地后的真实代码。

因此本文件作 **roadmap 索引 + 拆分原则 + 进度追踪**，每个里程碑落地前才生成对应 plan。**当前唯一已写好的 plan 是 M0.0**：[2026-05-23-m0-0-profilecapability-skeleton.md](2026-05-23-m0-0-profilecapability-skeleton.md)。

## Plan 拆分与生成规则

每完成一个里程碑，按以下流程产生下一个里程碑的 plan：

1. **前置条件确认**：上一个里程碑的所有 CRITICAL 用例全绿；契约（schema / API）已落到代码并通过 git diff 可读
2. **读取设计文档**：subsystems_design 对应章节 + system_design.md §15.x（P1）/ §16.3（P2）/ §16.4（P3）
3. **调用 writing-plans skill** 生成 `docs/superpowers/plans/<milestone-id>-<topic>.md`
4. **plan 自检**：spec coverage / placeholder / type consistency
5. **执行 handoff**：subagent-driven 或 inline

**禁止跨里程碑提前写 plan**：M0.1 落地前不写 M0.2 plan，因为 M0.1 的 schema 字段决定 M0.2 的 SQLite 表结构。同理 P1 未冻结前不写 P2.a。

**例外**：若某个里程碑出货项明显独立（如 P3.b 的迁移脚本不依赖 P3.a 的 Retrieval Planner），可并行起草。
